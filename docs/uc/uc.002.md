# UC.002 Аутентификация пользователя
<!-- Подробное описание сценария использования системы с привязкой к ролям участников и задействованным бизнес-сущностям 
https://confluence.mts.ru/pages/viewpage.action?pageId=375782119 
-->
| Название | Аутентификация пользователя |
|:---------------------------|:------|
| Описание | Пользователь хочет работать в системе |
| Участники | Пользователь с ролью: **Пользователь**, **Докладчик**, **Рецензент**, **Слушатель**, **Менеджер**, **Оператор** или **Техническая поддержка**, Frontend, Backend |
| Триггер | Пользователь нажал на кнопку **Sign In** |
| Предусловия | Пользователь зарегистрирован в системе, но не прошел процедуру аутентификации |
| Постусловия | Пользователь прошел процедеру аутентификации. Пользователь получил доступ к системе в соответствии с ролевой моделью. |
| Успешный сценарий | 1. Пользователь заполняет поле **Введите email**. 2. Пользователь заполняет поле **Пароль**. 3. Пользователь нажимает на копку **Sign In**. 4. Frontend валидирует корректность введенных данных. 6. Frontend отправляет данные на Backend. 7. Backend проверяет наличие полученных данных в таблице UsersAuth. 8. Backend возвращает на Frontend HTTP 200 OK. 9. Frontend отображает приложение пользователю в соответствии с его ролью в системе. |
| Альтернативный сценарий 1 | **6. Если пользователь заполнил одно из полей или оба поля некорректно.** 6.1. Frontend подсвечивает все некорректно заполненные поля красным цветом с описанием требований к заполнению каждого некорректного поля. 6.2. Пользователь вносит изменения в ранее некорректно заполненные поля. 6.3. Возвращение в пункт 3 основного сценария |
| Альтернативный сценарий 2 | **8. Если запись о пользователе не найдена.** 8.1. Backend возвращает на Fronend ошибку 404 Access Denied. 8.2. Frontend отображает страницу **Sign In** с ошибкой права доступа. _Завершение альтернативного сценария_ |
| Требования | *[SEC.001](docs\README.md)* |
